# 미션 내비게이션 (navigator)

인지(capture_fsm)와 위치추정(wall_localizer) 위에 얹히는 **주행/미션 계층**.
"어디로 가서, 무엇을 먼저 잡고, 언제 포기하고, 어떻게 보관함에 넣는가"를 결정해
`/cmd_vel` 을 낸다. 미션 로직은 ROS 없이 `sim_mission.py` 로 WSL에서 통째로 검증한다
(wall_localizer 와 같은 개발 방식).

## 룰 요약 (2026-07-14 공지)

- 4×4m, 담장 30cm. 배치 지점 42개(상하좌우 50cm 간격), 세트1 16개 + 세트2 12개가
  **경기장에 함께 배치**, 랜덤, 물체 높이 눕혀서 8cm, 스타트존에서 1m 이상.
- **세트2 과일면(룰 2026-07-15)**: 윗면 + 옆면 '서로 반대편 2면'에 부착, 일반적인
  형태(안 깎인·정상 색)라 어느 방향에서 접근해도 웬만하면 과일이 보인다. 시뮬은
  옆 반대편 2면(±65° 가시=커버리지 260%)으로 모델링 — `FRUIT_LAYOUT` 로 구 룰
  (random3)과 A/B 가능. 이 배치가 세트2 탐지를 크게 쉽게 하고 cube 인증을
  건전하게 만든다(아래 10번).
- **경기는 총 3분 한 번** — 그 안에 세트1 목표 형상 4개(+10×4=40) 와 세트2 목표
  과일 3개(+20×3=60) 를 모두 노린다 (합산 만점 100, 동점 시 완료 시간 순).
- 스타트존 우하단 40cm, 보관함 좌하단 40cm (태극기 스티커, 두 면에 3mm 테두리).
  위에서 봐서 보관함 경계 안에 완전히 들어가야 인정.
- 목표 형상+목표 과일은 경기 직전 공지 (1분 세팅 후 손 못 댐).
- **오픽업은 2배 감점(−20/−40)** — 모호하면 안 잡는 것이 항상 이득.
- 시간 예산 실측은 `sim_speed_sweep.py` (`--mode both` 가 실제 경기 형식;
  set1/set2/set1cube 단독 모드는 진단용).

## 전략 (구현된 정책)

1. **시간이 진짜 제약**: 모터 실측 전 추정 순항 0.15m/s 로 물체당 왕복 40~60초
   → 3분에 3~5개가 현실적 상한. 그래서:
   - **가치/시간 탐욕 선택**: 세트2(20점) 우선, 왕복 예상시간으로 나눈 점수율로 선택.
   - **시간 컷오프**: 남은 시간에 못 끝낼 목표는 시작하지 않음 + 진행 중에도 1초마다
     재검사(TIME_ABORT). 적재 상태로 경기 종료를 맞는 낭비를 없앤다.
2. **투어 우선**: 격자 행 사이 중간선(y=1.0, 3.0)을 왕복하며 측면 search 캠으로 전
   구역 분류 시도 → 확정 타깃이 생기면 즉시 중단하고 포획 트립. 미확인 물체 조사는
   투어 1바퀴 후에만 (초반 조사 도박은 기대 점수가 낮음 — 시뮬레이션으로 확인).
3. **오픽업 방어 다층화**:
   - 인지의 verify 게이트/veto 를 절대 우회하지 않음. READY 없이는 물체 앞 28cm
     정지선(approach_stop_short)에서 멈춤.
   - **의도치 않은 포획**(비포획 상태에서 IR 켜짐) 즉시 후진 배출.
   - veto 판정은 `verify_range` 로 내 목표에 귀속되는지 확인 — 시야에 낀 다른
     물체 때문에 진짜 타깃을 블랙리스트하지 않는다.
   - 태극기 지오펜스: 보관함 주변(0.55m)으로 투영되는 관측 무시.
4. **라이다는 위치추정 전용** (스캔면이 8cm 물체 위를 지남) → 장애물 지도는 카메라
   관측 누적(단안 거리: 물체 높이 8cm 고정 이용) + A* 회피, 최후엔 스탈 감지
   (엔코더 대비 이동 부족 → 후진 + 임시 장애물 등록).
5. **하역**: 진입 레인 y=0.30 에서 -x 방향 직진, 물체 중심 x=0.12/0.22/0.31 순환
   (겹치면 서쪽 벽이 백스톱). 후진+위글로 분리 — 물체는 마찰로 남는다.
6. **운반 중 유실**(IR 끊김 0.8s): 이미 검증된 물체이므로 분류 게이트 없이 회수
   — 단 APPROACH(조향 서보)는 거친다 (기억 위치만 믿는 blind push 는 옆으로
   빠진 물체를 영영 못 잡는다).
7. **더블 캐리** (`double_carry`, 기본 on): 적재함 깊이 15cm = 안쪽 1개(IR 확인)
   + **입구에 1개 더 물고** 운반. A 포획 직후 근처(1.5m)의 확정 타깃 중 보관함
   방향으로 완만하게(≤70°) 이어지는 B 를 골라 verify 정렬 후 밀착. 입구 물체는
   센서 확인 불가(개루프)·후진/급회전(≥0.5rad/s) 시 이탈 → 운반 중 각속도 제한,
   실패해도 A 하역은 보장(B 는 abandon). **주의**: capture_fsm 이 1물체 에피소드
   모델이라 실물 적용에는 '적재 상태 2번째 VERIFY 에피소드' 지원 필요 (인지 측
   후속 작업).
8. **밀항 물체 방지** (`shed_spin`+`shed_gate`): 운반 중 입구에 스스로 붙은
   정체불명 물체가 보관함까지 따라 들어가면 조용히 −40. 하역 진입 직전, 의도한
   B 가 없고 **이번 운반에서 입구 슬롯이 매핑된 물체를 실제로 스쳤을 때만**
   제자리 고속 회전(1.1rad/s, 330°)으로 털어낸다(위험 게이트 `shed_gate`, 반경
   `stowaway_ride_radius=0.08`). A 는 빈 옆벽·뒷벽에 갇혀 안전.
   - ⚠ **게이트 근거(계측 2026-07-16)**: 셰드를 꺼도 밀항은 운반 회전에서 대부분
     자연 이탈해 100경기 하역단계 생존 0·오픽업 0. 무조건 돌던 구동작(단일 하역
     마다 ~5s)을 위험 게이트로 바꿔 명목 경기는 스핀 생략 → both +1~2·bothcube
     +5점, 오픽업·벽충돌 0 유지. 완전 제거(`shed_spin=False`)도 sim 상 동일
     안전이나, 실기 입구 마찰 불확실성 때문에 **게이트를 기본**으로 둔다.
   - **하역 정렬**(`deposit_align`): 도착점 제자리 회전(`rotate`, 기본)이 최선.
     후진 재접근(`reapproach`)은 앞이 열린 빈에서 후진이 적재물을 흘려 점수 폭락
     (both 30.6→16.9), 정렬 생략 관통(`push_through`)은 레인이 남벽 8cm 앞이라
     비정렬 진입이 벽충돌 폭증(708+/16경기)+오픽업 → **벽 근접 기하에서 정렬 필수**.
     두 대안은 실기 기하 재검토용 토글로만 남겨둠(검증 2026-07-16).
9. **엔드게임 헤일메리** (`hail_mary`, 기본 on): 채점은 종료 시점 보관함 안 물체만
   세므로 **버저 때 적재만 한 상태는 감점 0**. 시간 컷오프 때문에 아무 목표도
   시작할 수 없는 막판에는 PARK 로 시간을 버리는 대신, 컷오프를 무시하고 open
   확정 타깃(val/trip 최대)을 시도한다 — 완주하면 +10/+20, 못 하면 0 (기대손실 0).
   verify 게이트는 그대로 지나므로 오픽업 방어 불변(트립 수 비례 위험만 존재).
   헤일메리 목표는 TIME_ABORT 재검사를 건너뛴다(아니면 1초마다 재중단 루프).
   스윕(2026-07-16, 48시드 페어드 @0.15): **+5.2**(both+bothcube 합 68.8→74.0),
   벽충돌 0, 시나리오 E2 로 회귀 가드. 같은 스윕에서 기각된 것들:
   `standoff_dist` 0.50 단축(평균 +2.3 이나 seed102 재스쿱 무한루프 0점 —
   평균 뒤의 파국 실패 모드), 미확인 조기조사(first_lane −1.7/always −5.8/
   근거리 probe −2.5 — full_tour 재확인), 투어 1레인 축소(+0.6 노이즈),
   하역 가속 계열(벽충돌/스필 동반), retreat 단축(조합에서 bothcube 벽 74회).
   ⚠ 개별 승자도 조합하면 상쇄되는 카오스 상호작용 확인 — 파라미터 채택은
   메커니즘 있는 것만, 확증은 48시드+ 그리고 시나리오 스위트 필수.

10. **cube 공지 경기** (`target_set1: cube` → cube_hunt 모드 자동):
   - 무지면 큐브는 단일 시점으로 세트1/세트2 구분이 원리적으로 불가 (실물
     분류기도 무지면 세트2 큐브를 'cube'로 본다). **위치 기반 다시점 인증**으로
     해결: 물체별 무지면 관측의 방위 섹터(16분할)를 누적, 연속 갭 ≤ 90° + 과일
     관측 전무 → 세트1 확정. **신 룰이 이 판정을 건전하게 만든다**: 세트2 큐브는
     항상 옆 반대편 2면에 과일이라 무지 사각이 180° 떨어진 두 50° 창뿐 → 그런
     큐브의 blank 관측은 최소 130° 갭을 남겨 '모든 갭 ≤ 90°' 를 통과할 수 없다
     (set2 오인증 원리적 0).
   - 투어가 외곽 일주로 바뀌고, 섹터가 부족한 큐브는 갭 방향 **보완 방문**
     (관측만, 포획 없음). verify 의 READY 는 인증된 항목에서만 수락.
   - 외곽 열 물체의 벽쪽 면은 벽-물체 틈이 좁아 관측 불가 → 내측 물체만 인증
     가능. 기대치는 낮다(시뮬레이션 ~0-10점) — cube 공지 시 세트2(60점) 위주
     운영이 정답이고, 큐브는 인증된 것만 기회적으로.

## 파일

| 파일 | 역할 |
|---|---|
| `nav_core.py` | ROS-free: 자세 제어기, A*(10cm 격자+시선단축), 구역, 스탈 감지, 단안거리 |
| `mission_fsm.py` | ROS-free: 미션 상태기계 + 물체 기억 + 목표 선택/시간 예산 |
| `sim_mission.py` | WSL 검증: 3분 경기 전체 시뮬레이션, 시나리오 8종(A~H) |
| `replay_mission.py` | 리플레이: 경기를 기록해 위에서 본 인터랙티브 HTML 뷰어 생성 |
| `navigator_node.py` | ROS 2 노드: 토픽/UDP 입출력 + 20Hz 제어 루프 (로직 없음) |
| `params.yaml` | 노드 파라미터 (실측값으로 갱신할 것) |

미션 상태: `TOUR → GOTO → APPROACH → CAPTURE → TRANSPORT →
[DEPOSIT_SHED (밀항 위험 시)] → DEPOSIT_PUSH → DEPOSIT_RELEASE → (반복) →
PARK/DONE`, 이탈은 `RETREAT` 경유. (`deposit_align="reapproach"` 시 정렬은
`DEPOSIT_REALIGN` 경유 — 기본 아님.)

## 통합 엔진(merged) 호환성

인지가 per-set 2개 검출기/분류기에서 **검출기 1개 + 9클래스 분류기 1개**로 병합되면
(runtime/merged_pipeline.py, configs/combined_classes.py 정본), 관측이 실어 보내는
`set` 라벨의 의미가 바뀐다: `set_of("cube") = "set1"` 은 **라우팅용**이지 확정이
아니다 (set1 큐브와 과일 숨은 set2 큐브는 픽셀 동일). 통합 설계도 다시점 확인을
`mission_fsm._maybe_confirm_cube` 에 위임한다. 그래서 내비게이션은:

- **관측의 `set` 을 신뢰하지 않고 `cls` 에서 유도**한다 (`mission_fsm.derive_set`,
  정본 `combined_classes.CLASS_TO_SET` 를 import — 없으면 내장 별칭으로 폴백).
  축약형(`octa`)·정본 전체 이름(`octahedron`) 모두 수용.
- **`cube` 는 어느 set 도 아니다**(derive_set→None). 통합 엔진이 `{cls:"cube",
  set:"set1", state:"TARGET_CONFIRMED"}` 로 넘겨도 그 set/state 를 무시하고, 오직
  다시점 무지면 인증(`_maybe_confirm_cube`)만이 set1 큐브로 확정한다 → cube 타깃
  경기에서 set2 큐브를 확정 타깃으로 오인해 −40 나는 것을 막는다.
- 검증: `sim_mission.CUBE_SIGHTING_SET`("set1"=통합계약 vs None=구 per-set) 토글로
  A/B. **두 방출이 결과 완전 동일** + 전 모드 오픽업 0 → incoming set 을 실제로
  무시함이 증명됨. 목표 이름은 params.yaml 에서 정본과 일치시킬 것(`icosahedron`).

## WSL 검증

```bash
yolo/bin/python navigation/sim_mission.py            # 8개 시나리오 PASS 확인
yolo/bin/python navigation/sim_mission.py --seeds 20 # 명목 시드 확대

# 리플레이: 경기를 위에서 본 인터랙티브 HTML 로 (브라우저로 열기)
yolo/bin/python navigation/replay_mission.py --seed 5 --mode both
yolo/bin/python navigation/replay_mission.py --scan 0 14 --mode both  # 좋은 경기 고르기
```

시나리오: A 명목(멀티시드: 오픽업 0·벽충돌 0 필수), B IR 순간끊김 무시,
C 운반 중 유실→회수, D 포획 빗맞음→재시도, E 잔여 60초 시간 컷오프(hail off),
E2 헤일메리(기본 on — 막판 시도가 안전하고 컷오프-only 대비 점수 하락 없음),
F 위치추정 열화 시 감속/정지, G 더블 캐리(A+B 동시 하역), H 더블 슬립
(입구 물체 이탈 시 A 하역 보장).

현재 결과 (2026-07-14, 12시드): **전 시나리오(8종) PASS, 전 콤보 오픽업 0·벽충돌 0.**

과일 배치 룰 A/B (12시드, opposite2=현행 vs random3=구, @0.20):

| 콤보 | 구 룰(random3) | 신 룰(opposite2) | Δ |
|---|---|---|---|
| set2 @0.20 | 25.0점 (1.25) | **33.3점 (1.67)** | +8.3 |
| both @0.20 | 26.7점 (1.75) | **35.0점 (2.17)** | +8.3 |
| bothcube @0.20 | 35.0점 (1.75) | **40.8점 (2.08)** | +5.8 |
| set1cube @0.20 | 2.5점 (0.25) | 1.7점 (0.17) | ~0 (물리 병목) |

전 콤보 **오픽업 0·벽충돌 0**, 시나리오 8종 PASS. 만점(7개)은 하역당 ~23초 필요라
어떤 속도로도 불가 — 현실 목표 2~4개. 12시드 표준편차 ±5점이라 @0.15↔@0.20 순서
역전 등은 노이즈; 신뢰할 신호는 같은 속도에서 신 룰이 세트2/both/bothcube 를 **+6~8점**
올린다는 것(반대편 2면 → 과일 가시성↑ → micro-adjust 왕복↓). set1cube 는 외곽 열
큐브의 벽쪽 면이 관측 불가라 여전히 낮다. 수치는 `sim_speed_sweep.py` 로 재생성.

## Jetson 실행 (전체 스택, 터미널 5개)

```bash
# [1] 모터/오도메트리/IR
python3 motor_control/motor_bridge.py --ros-args --params-file motor_control/params.yaml
# [2] 라이다
ros2 launch sllidar_ros2 sllidar_c1_launch.py serial_port:=/dev/ttyUSB0
# [3] 위치추정
python3 localization/wall_localizer_node.py --ros-args --params-file localization/params.yaml
# [4] 인지 (UDP 브리지 켜기) — 목표는 경기 직전 공지로!
python3 deployment/run_perception.py --target apple --udp
# [5] 내비게이터 — 목표 공지 반영
python3 navigation/navigator_node.py --ros-args --params-file navigation/params.yaml \
    -p target_set1:=icosa -p target_set2:=apple
# 경기 시작 신호:
ros2 topic pub --once /mission_start std_msgs/msg/Empty {}
```

## 실기 전 캘리브레이션 체크리스트

1. **모터**: `calibrate_encoder.py` 로 `ticks_per_rev`·`max_wheel_speed` 실측 →
   `min_pwm`(바퀴가 돌기 시작하는 PWM) 실측 → `closed_loop: true` 벤치 검증 →
   `mission.cruise_v` 를 실측 최고속도의 ~75% 로.
2. **로봇 기하**: `robot_radius`(외접), `payload_offset`(중심→빈 포켓) 실측.
3. **조향 부호**: 물체를 카메라 우측에 두고 `steering.combined_offset_px` 부호 확인
   → `mission.steering_sign` 결정. **틀리면 접근이 전부 타임아웃 난다** (시뮬레이션에서
   재현됨 — 증상: APPROACH_TIMEOUT 반복).
4. **카메라 장착각**: `cam_yaws_deg` 실측 (search ±90° 기준).
5. **IR**: 빈에 물체 넣고 `/bin_ir` 토글 확인, 감지 거리 조정 (WIRING.md 1-4).
6. **투어 레인**: 실제 배치 격자 원점 확인 후 `tour_lanes_y` 조정 (행 사이 중간선).

## 남은 일 / 알려진 한계

- **세트1+세트2 동시 인지**: 룰상 두 세트가 한 경기에 같이 나오므로
  `run_perception.py` 가 두 파이프라인을 한 프로세스에서 프레임 교대로 돌려야 함
  (카메라는 이중 오픈 불가). 현재는 한 세트만 — 인지 쪽 후속 작업.
  내비게이터·UDP 프로토콜은 이미 양 세트를 수용한다 (`set` 필드).
- **`cube` 가 세트1 목표로 공지되는 경우**: 과일면이 안 보이는 세트2 큐브와 외형
  구분 불가. 인지의 `cube_target_min_confirmations`(다시점) 만으로는 확률적 —
  4면 이상 무지면 확인(궤도 선회) 로직은 미구현. 이 경우 세트2 위주 전략 권장.
- 하역 후 보관함 내 물체 재배치(밀림)는 시뮬레이션이 단순화함 — 실기에서 3개 이상
  넣을 때 앞선 물체가 경계 밖으로 밀리지 않는지 확인 필요.
- 상대 팀 없음(단독 경기)이라 동적 장애물 대응은 미구현.
